cyber.md

AI-native posture that speaks agent.

Developers and coding agents are now generating more code than security teams can realistically review by hand.

The old security workflow assumed humans opened pull requests, humans remembered the threat model, humans knew the dangerous parts of the codebase, and humans slowly reviewed everything before deployment.

That assumption no longer holds.

Coding agents are refactoring auth flows, changing infrastructure logic, updating dependencies, modifying API boundaries, and introducing new integrations independently. At the same time, attacker-operated agents are scaling reconnaissance, fuzzing, and exploit discovery faster than traditional AppSec processes can react.

We think repositories need a new primitive:

Versioned security posture that agents can read, diff, and apply during normal development.

We’re calling it cyber.md.

What is cyber.md?

cyber.md is an open Markdown convention for describing the defensive security posture of a repository in a format that coding agents can understand during development.

The file itself is intentionally simple. It defines:

  • protected assets
  • trust boundaries
  • security invariants
  • high-risk surfaces
  • preferred defensive patterns
  • required tests
  • review triggers
  • ownership expectations

The goal is not to teach agents how to attack the system.

The goal is to help them avoid weakening it while they write code.

Why Markdown?

Agents already understand Markdown extremely well.

Modern coding tools routinely consume repository-local files like:

README.md
CONTRIBUTING.md
AGENTS.md

cyber.md extends the same idea to security posture. A repository should expose enough durable defensive context so that agents can preserve important security properties during implementation, review, and refactoring.

A posture file, not a vulnerability report

cyber.md is not:

  • a pentest report
  • a risk register
  • an exploit notebook
  • a collection of bypass paths
  • a customer incident archive

The checked-in version should contain safe defensive guidance only. Commit the posture, not the exploit.

Example

# cyber.md

## Protected Assets
- Customer API tokens
- Tenant-scoped billing data
- Deployment credentials

## Trust Boundaries
### Public API Boundary
Invariant:
All external input must be authenticated, validated, and tenant-scoped before database access.

Required Tests:
- auth bypass regression test
- tenant isolation test
- rate limit coverage

## Sensitive Modules
### /services/auth
Additional Review Required:
- token generation
- session handling
- permission checks

Preferred Patterns:
- short-lived tokens
- centralized authorization helpers
- no inline permission logic

Not just for security review

Inside an agent-native workflow, cyber.md becomes a lightweight defensive context layer.

  1. inspect the task
  2. identify touched files
  3. classify relevant risk areas
  4. retrieve posture guidance
  5. inject a minimal advice packet into the coding session

Not a giant scan report. Not a 200-page policy document. Just relevant posture at the exact moment code is being written.

Under the hood

1. Posture generation

A generator scans the repository and produces or updates cyber.md.

2. Context retrieval

During development, a posture layer reads cyber.md via hooks, brokers, CI, bots, or IDE extensions.

3. Task-specific injection

Developer requests code change
        ↓
Task + files classified
        ↓
Relevant posture selected
        ↓
Sensitive context brokered if needed
        ↓
Small advice packet injected
        ↓
Agent writes product code
        ↓
Diff checked against invariants

Attackers can read Markdown too

A repository-visible posture file can expose sensitive surfaces and trust assumptions, so cyber.md must be treated as part of the trusted development surface.

It should never contain secrets, exploit recipes, bypass instructions, sensitive topology, customer-specific incidents, or offensive tradecraft.

Instruction poisoning is real

Files like AGENTS.md and cyber.md should be code reviewed, ownership protected, and diffed carefully.

Optional: split posture model

Public posture: Safe defensive guidance stored in Git.

Private posture: Sensitive context stored behind a broker and returned minimally per task.

What should go inside cyber.md?

Scope

  • repository
  • services
  • frameworks
  • languages
  • sensitivity level

Protected assets

  • credentials
  • identities
  • admin privileges
  • tenant data
  • integration secrets

Trust boundaries

  • invariant
  • ownership
  • validation expectations
  • required controls
  • review triggers

Security guidance

  • authorization expectations
  • tenant isolation rules
  • secrets handling
  • dependency hygiene
  • logging constraints
  • secure defaults
  • encryption requirements

Required tests

  • auth regressions
  • permission coverage
  • tenant isolation
  • input validation
  • audit logging
  • rate limiting

Why this matters

cyber.md shifts posture guidance earlier: into the exact moment an agent is about to write code.

  • invariants survive refactors
  • safer defaults spread naturally
  • regression tests accumulate
  • risky patterns get corrected earlier
  • agents become less likely to weaken critical paths

Getting started

Create cyber.md at the repository root. Start small and connect it to IDE hooks, pull request bots, CI checks, agent context injection, or MCP brokers.
